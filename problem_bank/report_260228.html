Context Engineering

컨텍스트 엔지니어링

모델의 능력은 동일했지만, 모델에게 주어지는 컨텍스트의 구조를 바꾸는 것만으로 0/4에서 10/10로 결과가 달라졌다. 결정적 전환점은 프롬프트 인스트럭션에서 Claude Skills로의 전환이었다.

🔒
컨텍스트 격리
각 에이전트는 독립된 컨텍스트 윈도우를 가짐
→ plan, write, validate, reflect 각각 별도 에이전트
📄
파일 기반 인터페이스
에이전트 간 통신은 구조화된 마크다운 파일로만 수행
→ problem.md → plan.md → solution.py → validate → 채점
✂️
최소 컨텍스트 원칙
각 에이전트는 자신의 작업에 필요한 최소 정보만 수신
→ write 에이전트는 plan.md만 읽고 코드 생성
📖
명시적 지식 외재화
암묵적 컨텍스트가 아닌 명시적 문서로 지식을 전달
→ 요약 대신 원본 마크다운 파일 전달
⚡
Claude Skills = 컨텍스트 엔지니어링의 실현
프롬프트 인스트럭션으로 에이전트를 정의하면 인스트럭션 자체가 컨텍스트를 소비하고 오염시킨다. Claude Skills는 에이전트의 행동 정의(SKILL.md)와 에이전트 간 데이터 전달을 모두 구조화된 파일로 외재화하여, 위 네 가지 원칙을 자연스럽게 달성한다.
🎛️ solve_coding_test (Supervisor) 🧩 get_coding_problem 📝 plan_coding_problem 💻 write_coding_problem 🛡️ validate_coding_problem 🔍 reflect_coding_problem

Skills Breakdown

Skill이란? — 각 에이전트가 무엇을 해야 하는지 정의한 구조화된 마크다운 파일(SKILL.md). 에이전트가 호출되면 해당 Skill의 내용이 그 에이전트의 컨텍스트에 주입되어 행동을 안내한다. 프롬프트 인스트럭션과 달리 파일로 외재화되어 있어, 메인 대화 컨텍스트를 오염시키지 않는 것이 핵심.
🎛️
solve_coding_test Supervisor
전체 풀이 루프를 오케스트레이션하는 최상위 Skill. problem.md가 존재하는지 확인 후, trial 1부터 최대 10회까지 Plan → Write → Validate → 채점 대기 → Reflect 루프를 관리. 핵심 규칙: 모든 서브 Skill 호출은 반드시 fresh subagent(Task tool)로 디스패치 — 이전 서브에이전트를 재사용하지 않고, 같은 Skill이라도 매 trial마다 새로운 에이전트를 생성하여 메인 컨텍스트를 작게 유지. 채점은 사용자에게 제출을 요청한 뒤 판정(Pass/WA/TLE/MLE/RE)을 기다리는 human-in-the-loop 방식.
🧩
get_coding_problem Problem Fetch
Playwright MCP로 SWEA 웹사이트에 접속해 문제 페이지를 탐색. 문제 설명, 입출력 형식, 제약 조건, 샘플 테스트 케이스를 추출하여 problem.md로 저장. 다음 에이전트는 이 파일만 읽으면 된다.
📝
plan_coding_problem Plan
problem.md를 읽고 알고리즘을 설계. 코드는 절대 작성하지 않고, 문제 분석 → 핵심 관찰 → 알고리즘 선택 및 근거 → 의사코드 수준 구현 단계 → 엣지 케이스 → 복잡도 분석까지 사고 과정만 정리하여 plan_{trial}.md로 저장. PyPy 3.7 환경의 시간·메모리 제약을 고려해 실현 가능성까지 검증.
💻
write_coding_problem Write
problem.md와 plan_{trial}.md를 읽고 Python 3.7(PyPy) 코드를 구현. plan의 알고리즘을 충실히 따르되, SWEA I/O 패턴(T = int(input()) 루프)에 맞춰 작성. 완성된 코드를 solution_{trial}.py로 저장한 뒤 샘플 테스트를 실행하여 통과할 때까지 수정·재테스트 반복.
🛡️
validate_coding_problem Validate
Write 에이전트가 생성한 코드를 제출 전에 정적 검증. 금지된 import(sys, io, os), 출력 형식, Python 3.7 문법 호환성 등을 사전 체크하여 RE/WA를 제출 전에 차단. 3차 아키텍처에서 추가된 중간 단계.
🔍
reflect_coding_problem Reflect
채점 실패 시 발동. problem.md와 모든 이전 plan·solution 히스토리를 읽어 같은 실수를 반복하지 않도록 함. 판정 유형(WA/TLE/MLE/RE)에 따라 진단 초점을 달리하고, 무엇이 틀렸는지 → 왜 틀렸는지 → 이전 plan이 놓친 것을 분석하여 수정된 plan_{next_trial}.md를 생성. 코드는 작성하지 않음.
1차 → 2차 교훈
"분리하라"
단일 에이전트의 실패 후 서브에이전트로 분리. 프롬프트 인스트럭션으로 에이전트를 정의하고 요약으로 정보를 전달했으나, 알고리즘의 정교한 세부사항이 손실됨.
2차 → 3차 교훈
"Skills로 전환하라"
프롬프트 인스트럭션을 Claude Skills(SKILL.md)로 대체. 에이전트의 행동 정의와 에이전트 간 정보 전달 모두 파일 기반으로 전환하여 컨텍스트 부패와 정보 손실을 동시에 해결 — 컨텍스트 엔지니어링의 진수.

프롬프트 엔지니어링이 "모델에게 무엇을 말할 것인가"의 문제라면,
컨텍스트 엔지니어링은 "모델에게 어떤 구조로 정보를 제공할 것인가"의 문제다.

프롬프트 인스트럭션으로 에이전트를 정의하면 인스트럭션 자체가 컨텍스트를 오염시킨다. Claude Skills는 에이전트의 행동 정의를 구조화된 파일로 외재화하여 이 문제를 근본적으로 해결한다.

같은 모델, 같은 문제, 같은 Reasoning Effort에서도 컨텍스트 구조만 바꾸면 결과가 근본적으로 달라진다. D7(VH) 난이도 문제를 포함해 10/10 전문제 통과는 이 원칙의 실증.

Problem Details

문제별 상세 풀이

#23009 / VH2505 · D7(VH) · 정답률 6.67%
연속하는 겹치는 구간의 수
치환 분해 트리 + O(N log N)

풀이 아이디어

  • 순열의 연속 구간(consecutive interval)은 치환 분해 트리로 구조화
  • 교차하는 쌍은 선형 노드(ascending/descending)에서만 발생
  • 각 선형 노드에서 자식 크기의 접두사합 + 이진 탐색으로 O(k log k) 계산
Trial 1 — MLE (10/12 TC)
원인: 노드 풀 아키텍처에서 선형 노드 평탄화 시 기존 자식 리스트가 풀에 남아 해제되지 않음. 최악의 경우 pool_children에 O(N²) 참조 누적 → 256MB 초과.
Trial 2 — 통과 ✓
수정: 인라인 처리 방식으로 전환. 선형 노드를 부모에 흡수 시 즉시 기여값 계산 후 children_sizes 폐기(finalize-on-absorption). 전체 O(N) 메모리 유지.
#24424 / VH2521 · D6(VH) · 정답률 23.53%
올바른 경로
CFL-도달성 Dijkstra

풀이 아이디어

  • CNF 형태의 CFG와 방향 그래프에서 문맥 자유 언어 제약 최단 경로
  • 상태: (비단말 X, 출발 u, 도착 v) — 최대 26³ = 17,576개
  • 터미널 규칙으로 거리 1인 초기 상태 힙 삽입, 이진 규칙은 상태 확정 시 조인
Trial 1 — 통과 ✓
상태 공간이 17,576개로 작아 PyPy에서 충분히 빠름. 1차 시도에서 바로 통과.
#24529 / VH2525 · D6(VH) · 정답률 7.69%
격자판 위의 달팽이
그리디 + Bigint 비트셋 배낭 DP

풀이 아이디어

  • (1,1)과 (2,N)에 가장 작은 두 값 배치 (필수 경유지)
  • 나머지 2(N-1)개 값을 두 그룹 F(윗행), L(아랫행)로 균등 분할 → 합의 차이 최소화
  • F 오름차순, L 내림차순 배치 → 산 모양 배열로 윈도우 합 보장
  • 균형 분할은 Python bigint 비트셋 배낭 DP
Trial 1 — WA (18/47 TC)
원인: L 그룹을 오름차순으로 정렬 → 전환 윈도우에서 양쪽의 가장 작은 값이 집중.
Trial 2 — 통과 ✓
수정: L.sort(reverse=True) — 한 줄 수정.
#24702 / VH2530 · D7(VH) · 정답률 30.00%
2의 거듭제곱
p-adic 리프팅 이산 로그

풀이 아이디어

  • f(N) = 2^N mod 10^K의 마지막 K자리에 X를 포함시킬 목표값 T를 CRT로 구성
  • 2는 mod 5^K의 원시근 → 이산 로그 항상 존재
  • Hensel-style p진 리프팅으로 O(K) 단계만에 로그 추출
Trial 1 — RE
원인: import io SWEA 환경에서 금지.
Trial 2 — WA (4,400/12,190 TC)
원인: K=71 사용 시 N 값이 ~10^49까지 커져 채점기의 64비트 정수 범위 초과 추정.
Trial 3 — 통과 ✓
수정: K를 71→20으로 축소. N 최대값 ~1.53×10^14로 64비트 내 수용.
