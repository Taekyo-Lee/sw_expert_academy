## 1. 핵심 발견: 컨텍스트 엔지니어링

### 1.1 한 줄 요약

> **프롬프트 엔지니어링**이 "모델에게 *무엇을* 말할 것인가"의 문제라면,
> **컨텍스트 엔지니어링**은 "모델에게 *어떤 구조로* 정보를 제공할 것인가"의 문제다.

### 1.2 실험이 증명한 것

동일한 모델(Claude Opus 4.6), 동일한 문제, 동일한 Reasoning Effort에서 **에이전트 아키텍처(= 컨텍스트 구조)만 변경**했을 때 결과가 근본적으로 달라졌다.

이것은 "더 좋은 프롬프트를 쓰자"가 아니라, **"정보를 어떻게 구조화하여 모델에게 전달하느냐"가 성능의 결정적 요인**이라는 것을 의미한다.

## 2. 아키텍처 진화: 세 번의 시도

### 2.1 진화 요약

| 시도 | 아키텍처 | 결과 | 실패 원인 |
|------|---------|------|----------|
| 1차 | 단일 에이전트 (모든 작업을 하나의 컨텍스트에서) | **코드 생성 불가** | 컨텍스트 부패(Context Rot) |
| 2차 | 프롬프트 기반 서브에이전트 (요약으로 정보 전달) | **1/4 통과** | 정보 손실(Information Loss) |
| **3차** | **Claude Skills 기반 서브에이전트 (파일로 정보 전달)** | **10/10 통과** | — |

### 2.2 1차 시도: 단일 에이전트

**구조:** plan → write → reflect를 하나의 대화(컨텍스트 윈도우) 안에서 반복

**문제: 컨텍스트 부패(Context Rot)**
- 대화가 길어지면서 초반에 제공된 문제 설명, 알고리즘 설계 정보가 attention에서 밀려남
- 모델이 점점 코드 생성 능력을 상실 — 결국 유효한 코드를 만들지 못하는 상태에 도달
- 이는 모델의 능력 문제가 아닌, **컨텍스트 관리** 문제

**교훈:** "에이전트를 분리해야 한다"

### 2.3 2차 시도: 프롬프트 기반 서브에이전트

**구조:** Problem Fetch → Plan → Write → Reflect를 별도 에이전트로 분리. 각 에이전트의 출력을 **요약하여** 다음 에이전트의 프롬프트에 주입

**문제: 정보 손실(Information Loss)**
- 에이전트 정의: 프롬프트 인스트럭션으로 동작 지시 → 인스트럭션 자체가 컨텍스트를 소비
- 정보 전달: 이전 에이전트 출력을 요약 → 알고리즘의 정교한 세부사항(엣지 케이스 처리, 자료구조 선택 근거 등)이 누락
- 결과: 4문제 중 1문제만 통과

**교훈:** "에이전트를 분리하는 것만으로는 부족하다. 정보 전달 방식이 핵심이다"

### 2.4 3차 시도: Claude Skills 기반 서브에이전트 (최종)

**구조:** 각 에이전트를 **Claude Skills(SKILL.md 파일)** 로 정의. 에이전트 간 통신은 **구조화된 마크다운 파일**로만 수행

**왜 성공했는가:**

| 변경 포인트 | 2차 (실패) | 3차 (성공) |
|------------|-----------|-----------|
| 에이전트 정의 방식 | 프롬프트 인스트럭션 주입 | **SKILL.md 파일** (컨텍스트 비소비) |
| 정보 전달 방식 | 이전 에이전트 출력을 요약 | **원본 마크다운 파일을 직접 읽기** (무손실) |
| 컨텍스트 격리 | 부분적 (인스트럭션이 공간 차지) | **완전 격리** (각 에이전트는 깨끗한 컨텍스트에서 시작) |

**결과:** D7 난이도 포함 10문제 전수 통과

## 3. 컨텍스트 엔지니어링 4원칙

이 실험에서 도출한 4가지 설계 원칙:

### 원칙 1: 컨텍스트 격리 (Context Isolation)

각 에이전트는 독립된 컨텍스트 윈도우를 가진다. plan, write, validate, reflect 에이전트가 서로의 컨텍스트를 오염시키지 않는다.

> **적용:** Supervisor(solving-problem)가 서브에이전트를 호출할 때, 매번 fresh subagent를 생성. 같은 Skill이라도 이전 에이전트를 재사용하지 않음.

### 원칙 2: 파일 기반 인터페이스 (File-Based Interface)

에이전트 간 통신은 구조화된 마크다운 파일로만 수행한다.

> **적용:** `problem.md` → `plan_{n}.md` → `solution_{n}.py` → verdict → `plan_{n+1}.md`

### 원칙 3: 최소 컨텍스트 원칙 (Minimum Context Principle)

각 에이전트는 자신의 작업에 필요한 최소 정보만 수신한다.

> **적용:** writing-solution 에이전트는 problem.md와 plan.md만 읽고 코드를 생성. 이전 trial의 실패한 코드나 디버깅 로그는 전달하지 않음.

### 원칙 4: 명시적 지식 외재화 (Explicit Knowledge Externalization)

암묵적 컨텍스트(대화 흐름, 요약)가 아닌 명시적 문서(파일)로 지식을 전달한다.

> **적용:** "이전에 MLE가 났으니 메모리를 줄여라"가 아니라, diagnosing-failure가 구체적인 원인 분석과 수정 방향을 `plan_{n+1}.md`에 명시적으로 기록.

## 4. Skill 시스템 설계

### 4.1 설계 철학

**"작은 에이전트, 명확한 인터페이스, 상태 외재화"**

| 원칙 | 적용 | 효과 |
|------|------|------|
| **단일 책임** | plan / write / validate / stress / reflect 각각 독립 Skill | 한 에이전트가 한 가지 작업에만 집중 |
| **파일 기반 통신** | `plan.md` → `solution.py` → `verdict` | 컨텍스트 재사용 없이 깨끗한 시작 보장 |
| **진행 상태 추적** | `progress.md`로 모든 시도 기록 | `--resume`으로 중단 지점부터 재시작 가능 |
| **검증 단계 분리** | 제출 전 정적 검증 + 스트레스 테스트 | RE, 출력 형식 오류, TLE 사전 차단 |

### 4.2 Skill 구성 및 동작 흐름

```
┌─────────────────────────────────────────────────────────────────────┐
│  solving-problem (Supervisor)                                       │
│  전체 루프 오케스트레이션 · 최대 10 Trial · progress.md 관리         │
└────────────────────────────────┬────────────────────────────────────┘
                                 │
    ┌────────────────────────────┼────────────────────────────────────┐
    ▼                            ▼                                    ▼
┌──────────┐  problem.md   ┌──────────┐  plan_{n}.md  ┌────────────────┐
│ fetching │ ─────────────▶│ planning │ ────────────▶ │    writing     │
│ -problem │               │-solution │               │   -solution    │
└──────────┘               └──────────┘               └───────┬────────┘
                                                              │
                                                    solution_{n}.py
                                                              │
                                                    ┌─────────▼────────┐
                                    ┌───────────────│   validating     │
                                    │  FAIL → Write │   -solution      │
                                    │  재시도        └─────────┬────────┘
                                    │                         │ PASS
                                    │               ┌─────────▼────────┐
                                    │               │  stress-testing  │
                                    │               └─────────┬────────┘
                                    │                         │ PASS
                                    │               ┌─────────▼────────┐
                                    │               │   submitting     │
                                    │               │   -solution      │
                                    │               └─────────┬────────┘
                                    │                         │
                                    │              Pass? ─────┼───── Fail?
                                    │                ▼        │        ▼
                                    │            완료 (✅)     │  ┌──────────┐
                                    │                         │  │diagnosing│
                                    │                         │  │ -failure │
                                    │                         │  └────┬─────┘
                                    │                         │       │
                                    │                         │  plan_{n+1}.md
                                    │                         │       │
                                    └─────────────────────────┘       │
                                              ▲                       │
                                              └───── 다음 Trial ──────┘
```

### 4.3 각 Skill의 역할

| Skill | 역할 | 입력 | 출력 |
|-------|------|------|------|
| **solving-problem** | Supervisor. 전체 루프 오케스트레이션 | problem_id, 옵션 | progress.md |
| **fetching-problem** | Playwright로 문제 수집 | problem_id | problem.md |
| **planning-solution** | 알고리즘 설계 (코드 작성 금지) | problem.md | plan_{n}.md |
| **writing-solution** | 코드 구현 + 샘플 테스트 | problem.md, plan_{n}.md | solution_{n}.py |
| **validating-solution** | 정적 검증 (금지 import, 문법, 출력 형식) | solution_{n}.py | PASS / FAIL |
| **stress-testing** | 최대 입력 생성 + 시간/메모리 측정 | solution_{n}.py, problem.md | PASS / WARNING / FAIL |
| **submitting-solution** | Playwright로 채점 제출 | solution_{n}.py | 판정 결과 |
| **diagnosing-failure** | 실패 원인 분석 + 수정 계획 수립 | 판정, 전체 히스토리 | plan_{n+1}.md |

## 5. 검증 파이프라인: "제출은 골든 타임이다"

문제당 최대 10회 제출 제한이 있으므로, **로컬에서 최대한 문제를 잡아내는 것**이 핵심 전략이다. 3차 아키텍처에서 추가한 **validating-solution**과 **stress-testing** 단계가 결정적이었다.

### 5.1 4단계 검증

| 단계 | 검증 내용 | 실제로 잡은 오류 |
|------|----------|----------------|
| **1. 샘플 테스트** | writing-solution에서 샘플 입출력 실행 | 기본 구현 오류 |
| **2. 정적 검증** | 금지 import, Python 3.7 문법, 출력 형식 | `import io` (RE), `:=` 사용 (구문 오류), `#T` 접두사 오류 (WA) |
| **3. 스트레스 테스트** | 최대 크기 입력에서 시간/메모리 측정 | O(N^2) DP의 TLE, 노드 풀의 MLE |
| **4. 채점 제출** | 최종 채점 | 예상치 못한 경계 케이스 |

### 5.2 검증 없이 제출했다면?

| 문제 | 검증 단계에서 잡은 오류 | 제출 기회 절약 |
|------|----------------------|--------------|
| #24702 | `import io` → RE | 1회 |
| #25841 | `import io`, `import os` → RE | 1회 |
| #25660 | `#T` 출력 형식 오류 → WA 0점 | 1회 |

> 정적 검증만으로 3회의 제출 기회를 절약. 10회 제한 환경에서 이 차이는 치명적이다.

## 6. SWEA 채점 환경: 시행착오로 배운 제약

**"채점 환경의 제약은 문서화보다 시행착오로 발견된다"**

### 6.1 금지 사항 (RE 유발)

```python
# 금지 — 사용 시 런타임 에러(RE)
import sys          # #24702 1차 제출에서 발견
import io           # #24702, #25841에서 발견
import os           # #25841에서 발견
x := expr           # walrus operator (Python 3.8+), 구문 오류
match x:            # match/case (Python 3.10+), 구문 오류
```

### 6.2 허용 사항

```python
# 허용
input() / print()                                    # 표준 I/O
collections, heapq, bisect, math, itertools          # 표준 라이브러리
from collections import defaultdict, deque, Counter  # 일반적 자료구조
```

### 6.3 성능 관련

| 항목 | 세부 |
|------|------|
| 런타임 | PyPy 3.7 (CPython 대비 3~10배 빠름) |
| Python 문법 | 3.7 이하만 사용 가능 |
| 출력 형식 | 매우 민감 — 공백, 줄바꿈 하나로 WA 발생 가능 |

## 7. 실패 모드별 대응 전략

**"실패는 피드백이다 — 패턴을 인식하고 전략을 전환한다"**

### 7.1 실패 유형별 진단 및 대응

| 실패 유형 | 진단 초점 | 대응 전략 | 실제 사례 |
|-----------|---------|----------|----------|
| **MLE** | 데이터 구조의 메모리 복잡도 | 자료구조 최적화 또는 알고리즘 변경 | #23009: 노드 풀 O(N^2) → 인라인 O(N) |
| **TLE** | 시간 복잡도 클래스 | 복잡도 클래스 하락 또는 알고리즘 근본 변경 | #26013: O(N^2) DP → O(N log N) NTT |
| **WA (일부 통과)** | 로직 오류 또는 경계 조건 | 실패 케이스 분석 + 소규모 반례 구축 | #24529: 정렬 방향 오류 (한 줄 수정) |
| **WA (0점)** | 출력 형식 또는 근본적 알고리즘 오류 | 출력 형식 검증 우선 → 알고리즘 재검토 | #25660: 지수적 폭발 + 출력 형식 동시 오류 |
| **RE** | 금지 import 또는 예외 미처리 | 정적 검증 강화 | #24702, #25841: import io/os |

### 7.2 Stuck Detection (교착 감지)

diagnosing-failure Skill에 내장된 패턴 분석:

| 패턴 | 감지 조건 | 조치 |
|------|----------|------|
| **Plateau** | 3회 이상 같은 verdict + 같은 알고리즘 최적화만 반복 | **STUCK 선언** → 근본적 알고리즘 변경 강제 |
| **Oscillation** | WA → TLE → WA 식으로 verdict가 진동 | 두 제약을 동시 만족하는 새 알고리즘 필요 |
| **Regression** | 이전보다 통과 TC 수가 감소 | 이전 trial의 장점을 보존하며 수정 |

> #25478(수신기 배정)은 4회 시도 끝에 통과. MLE → WA(93/140) → WA(139/140) → Pass 과정에서 매 trial마다 알고리즘의 근본 변경이 필요했고, diagnosing-failure가 "같은 방향의 최적화"를 반복하지 않도록 가이드함.

## 8. D7 문제 접근법: Iterative Refinement

**"D7 문제는 한 번에 풀리지 않는다"**

### 8.1 통계

| 지표 | 값 |
|------|---|
| D7 문제 수 | 9문제 |
| 1회 통과 | 1문제 (#25699 타일링, 정답률 16.67%) |
| 평균 시도 | 2.3회 |
| 최다 시도 | 4회 (#25478 수신기 배정, 정답률 7.41%) |

### 8.2 대표 사례: #25478 수신기 배정 (4 Trial)

| Trial | 결과 | 원인 | 해결 방향 |
|-------|------|------|----------|
| 1 | MLE | 2^N 비트마스크 (N=50) → 메모리 폭발 | 알고리즘 근본 변경: 고유 구간 그래프로 전환 |
| 2 | WA 93/140 | 단순 LBFS가 유효한 순서를 보장하지 못함 | LBFS+ 적용 |
| 3 | WA 139/140 | N>=20에서 타이브레이킹 실패 | 우선순위 기반 LBFS+ + 분할 개선 |
| 4 | **Pass** | — | 4,211건 스트레스 테스트 통과 후 제출 |

> **핵심:** Trial 1→2에서 알고리즘 자체를 변경(비트마스크 → 구간 그래프), Trial 2→3→4에서 같은 알고리즘 내 점진적 개선. diagnosing-failure가 매 trial마다 "이전에 시도한 것"을 히스토리로 참조하여 같은 실수를 반복하지 않도록 유도.

### 8.3 Retry Policy

| 단계 | 실패 시 동작 | 최대 재시도 |
|------|-------------|-----------|

## 10. Takeaways

### 10.1 AI 에이전트 설계 시 기억할 것

1. **컨텍스트는 유한하고, 부패한다.** 긴 대화에서 모델의 성능이 떨어진다면, 프롬프트가 아니라 컨텍스트 구조를 의심하라.

2. **요약은 손실이다.** 에이전트 간 정보를 요약하여 전달하면, 알고리즘 세부사항처럼 정밀한 정보가 누락된다. 원본 파일을 직접 읽게 하라.

3. **에이전트는 작을수록 좋다.** 한 에이전트에 여러 역할을 부여하면 컨텍스트가 오염된다. "계획만 세우는 에이전트", "코드만 쓰는 에이전트"로 분리하라.

4. **검증은 제출 전에.** 제출 횟수 제한이 있는 환경에서는 로컬 검증 파이프라인이 경쟁력의 원천이다.

5. **실패 히스토리를 구조화하라.** diagnosing-failure가 모든 이전 plan과 solution 히스토리를 읽는 것이, 같은 실수를 반복하지 않는 핵심이다.

### 10.2 팀에 도입할 때

| 항목 | 설명 |
|------|------|
| `.env` 파일 | `SWEA_BASE_URL`, `SWEA_ID`, `SWEA_PW` 설정 |
| Playwright MCP | fetching-problem, submitting-solution에 필요 |
| Python 환경 | 로컬 테스트는 Python 3.10, 제출은 PyPy 3.7 호환 필요 |

| 모드 | 용도 | 명령어 |
|------|------|--------|
| **Manual** | 사용자가 직접 제출하고 결과를 에이전트에 전달 | `/solving-problem <id> --manual` |
| **Auto** | 브라우저 자동 제출 + 결과 수집 | `/solving-problem <id> --auto` |

### 10.3 한계 인식

| 영역 | AI의 역할 | 비고 |
|------|----------|------|
| 알고리즘 구현 | 매우 강함 | 이 시스템의 핵심 강점 |
| 실패 진단 및 개선 | 강함 | 히스토리 기반 반복 학습 |
| 문제 해석 및 전략 수립 | 보통 | 비표준적 문제에서는 사람의 개입이 필요할 수 있음 |
| 경계 케이스 발견 | 제한적 | 스트레스 테스트로 보완하나, 모든 경우를 커버하지 못함 |

> D7 이상의 문제는 평균 2~4회 trial이 필요하므로, 시간 여유를 확보하고 접근할 것을 권장.

## 11. 결론

이 실험의 핵심 메시지는 단순하다:

> **"같은 모델도 컨텍스트 구조에 따라 완전히 다른 결과를 만든다."**

| 원칙 | 적용 방법 |
|------|----------|
| 컨텍스트 격리 | 각 에이전트에 독립된 컨텍스트 윈도우 부여 |
| 파일 기반 인터페이스 | 에이전트 간 통신을 구조화된 파일로 한정 |
| 최소 컨텍스트 | 각 에이전트는 필요한 최소 정보만 수신 |
| 지식 외재화 | 요약이 아닌 원본 마크다운 파일을 직접 전달 |

프롬프트를 더 잘 쓰는 것이 아니라, **정보가 흐르는 구조를 설계하는 것**이 AI 에이전트 성능의 핵심이다.
