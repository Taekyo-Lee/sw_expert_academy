## 1. 아키텍처 설계: 컨텍스트 엔지니어링의 중요성

**"같은 모델, 같은 문제, 같은 추론 수준에서도 컨텍스트 구조만 바꾸면 결과가 근본적으로 달라진다"**

| 시도 | 아키텍처 | 결과 | 핵심 교훈 |
|------|---------|------|----------|
| 1차 | 단일 에이전트 (plan→write→reflect in one context) | ❌ 코드 생성 불가 | **컨텍스트 부패(context rot)**: 긴 대화에서 초반 정보가 attention에서 밀려나 모델의 코드 생성 능력 상실 |
| 2차 | 요약 전달 서브에이전트 | ⚠️ 4문제 중 1문제 통과 | **정보 손실**: 이전 에이전트의 출력을 요약해 전달하면 알고리즘의 정교한 세부사항이 누락됨 |
| **3차** | **파일 기반 스킬 분리** | ✅ **10/10 통과** | **무손실 전달 + 컨텍스트 격리**: 각 에이전트가 독립된 컨텍스트에서 마크다운 파일만 읽고 작업 |

> **프롬프트 엔지니어링**이 "모델에게 무엇을 말할 것인가"의 문제라면,
> **컨텍스트 엔지니어링**은 "모델에게 어떤 구조로 정보를 제공할 것인가"의 문제이다.

## 2. Skill 시스템 설계 철학

**"작은 에이전트, 명확한 인터페이스, 상태 외재화"**

| 원칙 | 적용 | 효과 |
|------|------|------|
| **단일 책임** | plan / write / validate / stress / reflect 각각 독립 skill | 한 에이전트가 한 가지 작업에만 집중, 품질 향상 |
| **파일 기반 통신** | `plan.md` → `solution.py` → `verdict` | 컨텍스트 재사용 없이 깨끗한 시작 보장 |
| **진행 상태 추적** | `progress.md`로 모든 시도 기록 | `--resume`으로 중단 지점부터 재시작 가능 |
| **검증 단계 분리** | `validating-solution`으로 제출 전 정적 검증 | RE, 출력 형식 오류 등 사전 차단 |

## 3. SWEA 채점 환경 대응

**"채점 환경의 제약은 문서화보다 시행착오로 배운다"**

| 함정 | 발생 문제 | 발견 경로 | 해결책 |
|------|----------|----------|--------|
| `import sys` 금지 | RE (런타임 에러) | 24702 1차 제출 | `input()` / `print()`만 사용 |
| `import io`, `import os` 금지 | RE | 24702, 25841 | 순수 표준 입력출력 고수 |
| `:=` (walrus operator) 금지 | 구문 오류 | static validation | Python 3.7 문법 체크 |
| Python vs PyPy 속도 차이 | TLE 예측 실패 | stress testing | CPython 3.10 대비 PyPy가 3~10배 빠름 |
| 출력 형식 민감성 | WA 0/77 (25660) | sample 테스트 | problem.md의 샘플 출력도 검증 필요 |

### SWEA Python 3.7 제약 요약

```python
# ❌ 금지
import sys
import io
import os
# walrus operator (:=)
# match/case (Python 3.10+)

# ✅ 허용
input() / print()
collections, heapq, bisect, math, itertools 등 표준 라이브러리
```

## 4. 실패 모드별 대응 전략

**"실패는 피드백이다 — 패턴을 인식하고 전략을 전환한다"**

| 실패 유형 | 진단 | 전략 | 사례 |
|-----------|------|------|------|
| **MLE** | 메모리 복잡도 계산 오류 | 데이터 구조 최적화/알고리즘 변경 | 23009: 노드 풀 O(N²) → 인라인 O(N) |
| **TLE** | 시간 복잡도 초과 | 복잡도 클래스 하락 또는 상수 최적화 | 26013: O(N²) DP → O(N log N) NTT |
| **WA (일부 통과)** | 로직 오류 또는 경계 조건 | 디버깅 + 소규모 테스트 | 24529: 정렬 방향 오류 |
| **WA (0 통과)** | 출력 형식 또는 근본적 오류 | 출력 형식 검증 → 알고리즘 재검토 | 25660: 지수 폭발 + 형식 오류 |
| **RE** | 예외 처리 누락 또는 금지 import | static validation 강화 | 24702, 25841: import io/os |

## 5. 어려운 문제(D7) 접근법

**"D7 문제는 한 번에 풀리지 않는다 — iterative refinement가 핵심"**

```
Trial 1: 알고리즘 설계 및 구현
    ↓ (fail: MLE/TLE/WA/RE)
Trial 2: 스트레스 테스트/검증 실패 → 알고리즘 수정
    ↓ (fail: WA)
Trial 3: 반례 발견 → 로직 수정
    ↓ (pass)
```

### 25478 (수신기 배정) 사례

| Trial | 결과 | 원인 | 해결책 |
|-------|------|------|--------|
| 1 | MLE | 2^N 비트마스크 (N=50) | 고유 구간 그래프로 전환 |
| 2 | WA 93/140 | LBFS 순서 불완전 | LBFS+ 적용 |
| 3 | WA 139/140 | 타이브레이킹 미흡 | 우선순위 기반 개선 |
| 4 | ✅ Pass | — | 4,211건 스트레스 테스트 통과 |

### 핵심 교훈

- **최대 10회 제한**이 있으므로 효율적인 iteration 필요
- **Stuck Detection**: 3회 이상 같은 알고리즘 최적화만 반복되면 근본적 알고리즘 변경 필요
- **Stress Testing**: 최대 입력에서 TLE/RE를 사전에 발견하여 제출 기회 보존

### Retry Policy

| 단계 | 실패 시 동작 | 최대 재시도 |
|------|-------------|------------|
| Write | diagnosing-failure → 재작성 | 3회 |
| Stress Test | diagnosing-failure → 알고리즘 변경 | 3회 |
| Validate | Write 단계로 회귀 | 3회 |
| Submit (Judge Fail) | diagnosing-failure → Trial 증가 | 10 Trial |

## 6. 검증 파이프라인의 가치

**"제출은 골든 타임이다 — 로컬에서 모든 걸 잡아내자"**

| 단계 | 검증 내용 | 도구 | 실제로 잡은 오류 |
|------|----------|------|----------------|
| **Local Test** | 샘플 입출력 | `test_solution.py` | 기본 구현 오류 |
| **Static Validate** | 금지 import, Python 3.7 문법, 출력 형식 | `validating-solution` | `import io`, `:=` 사용, `#T` 접두사 오류 |
| **Stress Test** | 최대 입력에서 시간/메모리 측정 | `stress_test.py` | O(N²) DP의 TLE, 노드 풀의 MLE |
| **Submit** | 최종 채점 | `submitting-solution` | 예상치 못한 경계 케이스 |

> **"Validating-solution은 3차 아키텍처의 핵심 추가"**
> Write → Validate → Submit 순서로 RE/WA를 사전에 차단하여 제출 기회를 보존한다.

## 7. 팀 적용을 위한 권고사항

**"이 시스템을 팀에 도입할 때 고려할 점"**

### 7.1 인프라 준비

| 항목 | 설명 |
|------|------|
| `.env` 파일 | 계정 정보 관리 (보안) — `SWEA_BASE_URL`, `SWEA_ID`, `SWEA_PW` |
| Playwright MCP | auto 모드용 브라우저 자동화 설정 |
| `uv` 환경 | Python 3.10 (로컬 테스트용) — PyPy 3.7과의 문법 호환성 체크 |

### 7.2 워크플로우 선택

| 모드 | 용도 | 권장 상황 |
|------|------|----------|
| **Manual** | 사용자가 직접 제출하고 결과 보고 | 알고리즘 이해 중심 학습, 제출 전 자체 검토 필요 시 |
| **Auto** | 브라우저 자동 제출 및 결과 수집 | 대량 문제 해결, 빠른 iteration 필요 시 |

### 7.3 Skill 확장 방향

- 새로운 문제 유형 발견 시 `diagnosing-failure`에 패턴 추가
- 팀 내 자주 발생하는 오류 유형별 custom validation rule 추가
- 특정 알고리즘 유형(DP, 그래프, 기하 등) 전용 스트레스 테스트 생성기 추가

### 7.4 한계 인식

| 영역 | 역할 | 비고 |
|------|------|------|
| **알고리즘 구현** | AI 자동화 | ✅ 이 시스템의 강점 |
| **문제 해결 전략** | 인간의 영역 | ❌ 여전히 사람이 설계 필요 |
| **경계 케이스 발견** | 혼합 | AI가 못 잡는 경우도 있음 |

> D7 이상의 문제는 평균 2~4회 trial 필요 — 시간 여유 확보 권장

## 8. 결론

이 시스템의 핵심 가치는 **"컨텍스트 관리"**와 **"체계적인 검증"**에 있다. LLM의 능력을 최대한 활용하면서도, 컨텍스트 부패를 방지하고 실패로부터 빠르게 학습하는 구조를 만드는 것이 D7 난이도 문제를 10/10 통과할 수 있었던 비결이다.

| 핵심 원칙 | 설명 |
|----------|------|
| **컨텍스트 격리** | 각 에이전트는 독립된 컨텍스트 윈도우를 가짐 |
| **스킬 기반 인터페이스** | 에이전트 간 통신은 구조화된 마크다운 파일로만 수행 |
| **최소 컨텍스트 원칙** | 각 에이전트는 자신의 작업에 필요한 최소 정보만 수신 |
| **명시적 지식 외재화** | 암묵적 컨텍스트(대화 흐름)가 아닌 명시적 문서(파일)로 지식을 전달 |

> **"요약된 정보가 아닌, 원본 마크다운 파일을 직접 읽도록 설계하라"**
