## 에이전트 아키텍처 실험 이력

10/10 통과는 세 번째 아키텍처(스킬 기반 서브에이전트)에서 달성한 결과다. 에이전트 간 정보 전달 방식이 성능에 결정적 영향을 미침.

| 시도 | 아키텍처 | 추론 수준 (Reasoning Effort) | 결과 | 실패 원인 |
|------|----------|----------|------|-----------|
| 1차 | 단일 에이전트 (plan → write → reflect를 하나의 컨텍스트에서 수행) | High | 코드 생성 불가 | 컨텍스트 부패(context rot) — 긴 대화에서 모델이 코드 생성 능력 상실 |
| 2차 | 서브에이전트 분리, 이전 에이전트의 요약을 다음 에이전트에 전달 | Medium | 4문제 중 1문제 통과 | 요약 전달 시 정보 손실 — 서브에이전트 간 완전 분리 미달성 |
| **3차** | **서브에이전트 완전 분리 — 각 에이전트는 이전 에이전트가 생성한 마크다운 파일만 읽음** | **High** | **10문제 중 10문제 통과** | — |

**핵심 교훈:** 서브에이전트 간 정보 전달은 대화 컨텍스트가 아닌 **스킬 기반**이어야 함. 각 에이전트가 독립된 컨텍스트에서 마크다운 파일(problem.md, plan.md)만 읽고 작업하면 컨텍스트 부패 없이 안정적으로 동작.

## 컨텍스트 엔지니어링 (Context Engineering)

1차에서 3차까지의 아키텍처 변천은 곧 **컨텍스트 엔지니어링**의 과정. 모델의 능력은 동일했지만, 모델에게 주어지는 컨텍스트의 구조를 바꾸는 것만으로 0/4에서 10/10으로 결과가 달라졌음.

### 문제: LLM 컨텍스트 윈도우의 한계

대규모 언어 모델은 컨텍스트 윈도우라는 고정된 크기의 작업 기억(working memory)을 가짐. 난이도 D6-D7 수준의 알고리즘 문제를 풀려면 문제 이해 → 알고리즘 설계 → 구현 → 디버깅이라는 긴 사고 과정이 필요한데, 이 모든 것을 하나의 컨텍스트에 담으면 두 가지 문제가 발생:

1. **컨텍스트 부패 (Context Rot)**: 대화가 길어질수록 초반 정보가 주의(attention)에서 밀려나고, 모델의 출력 품질이 급격히 하락. 1차 시도에서는 설계까지는 성공했지만 코드를 생성하는 단계에서 모델이 능력을 상실.

2. **정보 간섭 (Context Pollution)**: 실패한 시도의 잔여 정보, 중간 탐색 과정, 부수적 논의 등이 컨텍스트에 누적되어 현재 작업에 간섭.

### 해결: 스킬 기반 컨텍스트 분리

컨텍스트 엔지니어링의 핵심은 **각 서브에이전트가 깨끗한 컨텍스트에서 시작**하도록 설계하는 것.

| 원칙 | 설명 | 적용 |
|------|------|------|
| **컨텍스트 격리** | 각 에이전트는 독립된 컨텍스트 윈도우를 가짐 | plan, write, static-validate, reflect 각각 별도 에이전트 |
| **스킬 기반 인터페이스** | 에이전트 간 통신은 구조화된 마크다운 파일로만 수행 | problem.md → plan.md → solution.py → static-validate → 채점 |
| **최소 컨텍스트 원칙** | 각 에이전트는 자신의 작업에 필요한 최소 정보만 수신 | write 에이전트는 plan.md만 읽고 코드 생성 |
| **명시적 지식 외재화** | 암묵적 컨텍스트(대화 흐름)가 아닌 명시적 문서(파일)로 지식을 전달 | 요약 대신 원본 마크다운 파일 전달 |

### 각 시도에서 배운 것

**1차 → 2차 교훈: "분리하라"**
단일 에이전트의 실패 후 서브에이전트로 분리. 그러나 이전 에이전트의 출력을 요약하여 다음 에이전트의 컨텍스트에 주입하는 방식을 택함. 이 요약 과정에서 정보가 손실되었고, 특히 알고리즘의 정교한 세부사항이 write 에이전트에 전달되지 못하였음.

**2차 → 3차 교훈: "파일로 전달하라"**
요약 전달을 폐기하고, 각 에이전트가 이전 에이전트가 생성한 마크다운 파일을 직접 읽도록 변경. 이로써:
- plan 에이전트의 알고리즘 설계가 100% 무손실로 write 에이전트에 전달됨
- reflect 에이전트의 실패 분석이 정확히 다음 plan에 반영됨
- 각 에이전트는 깨끗한 컨텍스트에서 출발하므로 컨텍스트 부패가 원천 차단됨

### 결론

> **프롬프트 엔지니어링**이 "모델에게 무엇을 말할 것인가"의 문제라면,
> **컨텍스트 엔지니어링**은 "모델에게 어떤 구조로 정보를 제공할 것인가"의 문제임.
>
> 같은 모델, 같은 문제, 같은 추론 수준에서도 컨텍스트 구조만 바꾸면 결과가 근본적으로 달라잠.
> D7 난이도 문제를 포함해 10/10 전문제 통과는 이 원칙의 실증.

## 23009 — 연속하는 겹치는 구간의 수 (통과 ✓)

**알고리즘:** 치환 분해 트리(Substitution Decomposition Tree) + O(N log N) 교차 쌍 세기

**풀이 아이디어:**
- 순열의 연속 구간(consecutive interval)은 치환 분해 트리로 구조화됨
- 교차하는 쌍은 선형 노드(ascending/descending)에서만 발생
- 각 선형 노드에서 자식 크기의 접두사합 + 이진 탐색으로 O(k log k) 계산

**1차 시도 (solution_1.py):** MLE — 12개 테스트 케이스 중 10개 통과 (2개 MLE)
- **원인:** 노드 풀(pool) 아키텍처에서 선형 노드 평탄화(flattening) 시 기존 자식 리스트가 풀에 남아 해제되지 않음. 최악의 경우(항등 순열) `pool_children`에 O(1 + 2 + ... + N) = O(N²) 참조가 누적되어 256MB 초과.
- **수정 (solution_2.py):** 영구적 트리 구축 대신 인라인 처리 방식으로 전환. 선형 노드를 다른 타입의 부모에 흡수될 때 즉시 기여값을 계산하고 `children_sizes`를 폐기(finalize-on-absorption). 같은 타입 흡수 시 기존 리스트에 in-place append하여 전체 O(N) 메모리 유지.

## 24424 — 올바른 경로 (통과 ✓)

**알고리즘:** CFL-도달성 상태 공간 위의 Dijkstra

**풀이 아이디어:**
- CNF 형태의 CFG와 방향 그래프에서 문맥 자유 언어 제약 최단 경로 문제
- 상태: `(비단말 기호 X, 출발 정점 u, 도착 정점 v)` — 최대 26³ = 17,576개
- 터미널 규칙으로 거리 1인 초기 상태를 힙에 삽입, 이진 규칙은 상태 확정 시 조인
- 정수 인코딩 힙 키, 플랫 `dist` 리스트, 지역 변수 캐싱으로 최적화

**1차 시도에서 통과.** 상태 공간이 17,576개로 작아 PyPy에서 충분히 빠름.

## 24529 — 격자판 위의 달팽이 (통과 ✓)

**알고리즘:** 그리디 + Python 정수 비트셋 배낭 DP

**풀이 아이디어:**
1. 모든 경로가 반드시 지나는 (1,1)과 (2,N)에 가장 작은 두 값 배치
2. 나머지 2(N-1)개의 값을 두 그룹 F(윗행), L(아랫행)로 균등 분할 → 합의 차이 최소화
3. F 오름차순, L 내림차순 배치 → 산 모양(mountain) 배열로 모든 슬라이딩 윈도우 합 ≥ min(sumF, sumL) 보장
4. 균형 분할은 Python bigint 비트셋 배낭 DP로 탐색

**1차 시도 (solution_1.py):** WA — 47개 테스트 케이스 중 18개만 통과
- **원인:** L 그룹을 오름차순(`L.sort()`)으로 정렬하여 산 모양 배열이 아닌 오름차순-오름차순 배열이 됨. 전환 윈도우에서 양쪽 그룹의 가장 작은 값이 집중되어 윈도우 합이 min(sumF, sumL) 미만으로 하락.
- **수정 (solution_2.py):** `L.sort()` → `L.sort(reverse=True)` 한 줄 수정.

## 24702 — 2의 거듭제곱 (통과 ✓)

**알고리즘:** 수론적 구성 — p진(p-adic) 리프팅을 이용한 이산 로그 계산

**풀이 아이디어:**
- f(N) = 2^N mod 10^K의 마지막 K자리에 X를 포함시킬 목표값 T를 CRT로 구성
- 2는 mod 5^K의 원시근 → 이산 로그 항상 존재
- Hensel-style p진 리프팅으로 O(K) 단계만에 로그 추출

**1차 시도 (solution_1.py):** RE — `import io` 금지
- **원인:** 빠른 I/O를 위해 `import io`를 사용했으나 SWEA 채점 환경에서 금지됨.

**2차 시도 (solution_2.py):** WA — 12,190개 테스트 케이스 중 4,400개만 통과
- **원인:** K=71 사용 시 N 값이 최대 ~10^49 (50자리)까지 커짐. 채점기의 검증 코드가 고정 너비 정수(64비트)를 사용하거나 문자열 길이 제한이 있을 가능성.
- **수정 (solution_3.py):** K를 71에서 20으로 축소. N 최대값이 ~1.53 × 10^14로 64비트 정수 범위 내에 수용됨. 오프셋도 50에서 8로 축소.

## 25333 — 시너지 효과 (통과 ✓)

**알고리즘:** Meet-in-the-middle + 포함-배제 + 행렬 트리 정리(Matrix-Tree Theorem)

**풀이 아이디어:**
- N개의 노드로 레이블 트리를 구성할 때, 의미 있는 노드의 "고립" 여부로 시너지 효과 계산
- g(t): 특정 t개의 의미 노드가 모두 고립될 때의 트리 수 — 등분할 라플라시안으로 닫힌 식 도출
- 포함-배제로 "정확히 고립" 카운트를 "최소 고립" 카운트로 변환
- Meet-in-the-middle로 부분집합 합 제약 하에서 가중 카운트 계산 (m ≤ 40)

**1차 시도 (solution_1.py):** TLE
- **원인:** N_r 계산 시 각 (r, rA, rB) 조합에 대해 모든 B-부분집합을 순회하며 이진 탐색 수행. 총 반복 횟수가 ~10^8-10^9에 달함.
- **수정 (solution_2.py):** g(t) × H(t) × N_r 계산을 단일 가중 부분집합 합 쿼리로 통합. 크기별 가중치 f(r)을 사전 계산하고, A-부분집합에 대한 접미사 가중합 배열을 구축하여 B-부분집합당 이진 탐색 1회로 축소.

## 25478 — 수신기 배정 (통과 ✓)

**알고리즘:** 고유 구간 그래프(Proper Interval Graph) 분해 + 간격 DP

**풀이 아이디어:**
- N개의 수신기를 X개의 사물함(1m 간격)에 배치. 거리 ≤ D이면 통신 가능.
- 통신 행렬 A를 만족하는 유효한 배치 수를 mod 10^9+7로 계산
- 연결 성분별로 고유 구간 그래프(proper interval graph) 구조를 이용
- LBFS+ 알고리즘으로 정준 순서(canonical ordering)를 추출하고, O(2^D) 상태의 간격 DP로 각 성분의 가능한 span별 배치 수를 계산
- 쌍둥이 클래스(twin class) 대칭성 처리 후, 성분 간 합성곱(convolution)으로 최종 답 산출

**1차 시도 (solution_1.py):** MLE
- **원인:** 2^N 비트마스크 DP로 N=50에서 2^50 상태 → 메모리 초과.
- **수정 (solution_2.py):** 비트마스크 제거, 고유 구간 순서 + O(2^D) 간격 DP로 전환.

**2차 시도 (solution_2.py):** WA — 140개 테스트 케이스 중 93개 통과
- **원인:** 단순 LBFS가 유효한 고유 구간 순서를 보장하지 못함.
- **수정 (solution_3.py):** 두 번째 스윕에 LBFS+ 알고리즘 적용.

**3차 시도 (solution_3.py):** WA — 140개 테스트 케이스 중 139개 통과
- **원인:** N≥20에서 LBFS+ 타이브레이킹 실패. 순서 검증은 통과하나 간격 DP에서 제약 불만족.
- **수정 (solution_4.py):** 우선순위 기반 LBFS+와 정제된 분할 개선(partition refinement) 구현. 4,211건 전수 스트레스 테스트 통과.

## 25660 — 산타의 선물 (통과 ✓)

**알고리즘:** 인기 매칭(Popular Matching) + 그리디 + 이분 매칭

**풀이 아이디어:**
- N명의 아이에게 N개의 선물을 배정. 과반수가 선호하는 다른 배정이 없는 "인기 매칭"이 존재하면 사전순 최소를 출력, 없으면 -1
- 2단계 매칭 구조: 1단계에서 1순위 선물을 받는 그룹(A)과, 나머지에서 비-S1 최상위 선물을 받는 그룹(B)
- 결정적 캐스케이드(deterministic cascade): 충돌 선물을 S1에 동시 추가하여 기하급수 분기 방지
- Hopcroft-Karp 이분 매칭으로 S1 커버리지 검증

**1차 시도 (solution_1.py):** WA — 77개 테스트 케이스 중 0개 통과
- **원인:** 지수적 백트래킹 폭발. k명이 동일 1순위를 공유할 때 2^k 분기 발생.
- **수정 (solution_2.py):** 결정적 캐스케이드 + 이분 매칭으로 교체.

**2차 시도 (solution_2.py):** WA — 77개 중 0개 통과 (출력 형식 오류)
- **원인:** problem.md의 샘플 출력에 잘못된 `#T` 접두사가 포함됨.
- **수정:** 출력 형식 수정 후 77/77 통과.

## 25699 — 타일링 (통과 ✓)

**알고리즘:** 격자 주기성(Lattice Periodicity) + 에르미트 정규형(Hermite Normal Form) 열거

**풀이 아이디어:**
- 단일 다각형(폴리오미노)의 평행이동 타일링으로 생성된 H×W 관찰 창이 주어짐. 타일의 최소 넓이를 구함.
- 타일링은 2D 정수 격자에 대응. 타일 넓이 = 격자의 행렬식(determinant)
- 모든 격자는 유일한 HNF 기저 (a, 0), (c, d)를 가짐. 넓이 = a×d
- 유효한 이동(translation) 테이블을 O(H²W²)로 사전 계산 후, 넓이 1부터 순차 열거하여 최초 유효 HNF 반환

**1차 시도에서 통과.** 정답률 4.76%의 문제를 단번에 해결.

## 25841 — 자매 결연 (통과 ✓)

**알고리즘:** 크루스칼 재구성 트리(Kruskal Reconstruction Tree) + 이진 리프팅 LCA

**풀이 아이디어:**
- 가중치 무방향 그래프에서 두 차량이 동시에 역방향으로 이동. 같은 도시나 같은 도로를 역방향으로 동시 사용 불가.
- 교환이 가능한 조건: 연결 성분이 단순 경로(simple path)가 아닐 때 — 양보할 "주차 공간"이 존재
- 크루스칼 재구성 트리로 간선 가중치 순서대로 성분을 병합하며 비-경로(non-path) 이벤트 추적
- 각 KRT 노드에서 `subtree_min_np`(하위 트리 내 최소 비-경로 이벤트 가중치)를 bottom-up 전파
- 쿼리 (X, Y)는 LCA를 찾아 `ans_weight(LCA)` 반환

**1차 시도 (solution_1.py):** RE — `import io`, `import os` 금지
- **원인:** 빠른 I/O를 위해 금지된 모듈 사용.
- **수정 (solution_2.py):** `input()` / `print()` 사용으로 교체. 알고리즘 변경 없음.

## 26013 — 비숍 (통과 ✓)

**알고리즘:** 페러스 보드 분해 정리(Ferrers Board Factorization) + NTT 합성곱

**풀이 아이디어:**
- N×N 체스판에 비공격 비숍 k개를 놓는 경우의 수를 k=1부터 2N-1까지 mod 998244353로 계산
- 대각선 패리티 분해: "/" 대각선과 "\" 대각선이 패리티에 따라 두 독립 그룹으로 분리
- 각 패리티 그룹에서의 비숍 배치 = 페러스 보드 위의 룩(rook) 배치 문제
- 페러스 보드 분해 정리에 의해 룩 다항식이 선형 인수의 곱으로 분해됨
- N×N 보드의 산술적 구조 덕분에 인수가 최대 2종류의 값만 취함
- 유한 차분(finite differences) + NTT 합성곱으로 O(N log N) 계산

**1차 시도 (solution_1.py):** TLE
- **원인:** O(N²) DP. N=100,000에서 ~10^10 연산으로 30초 제한 초과.
- **수정 (solution_2.py):** 페러스 보드 분해 + NTT로 O(N log N) 최적화. 50개 테스트 케이스 × N≤100K에서 충분히 빠름.
