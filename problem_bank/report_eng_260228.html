Context Engineering

Context Engineering

The model's capabilities were identical, yet simply changing the context structure shifted results from 0/4 to 10/10. The decisive turning point was switching from prompt instructions to Claude Skills.

🔒
Context Isolation
Each agent has an independent context window
→ Separate agents for plan, write, validate, reflect
📄
File-based Interface
Inter-agent communication only through structured markdown files
→ problem.md → plan.md → solution.py → validate → judge
✂️
Minimal Context Principle
Each agent receives only the minimum information needed for its task
→ Write agent reads only plan.md to generate code
📖
Explicit Knowledge Externalization
Knowledge transferred via explicit documents, not implicit context
→ Original markdown files instead of summaries
⚡
Claude Skills = Context Engineering Realized
Defining agents via prompt instructions means the instructions themselves consume and pollute the context. Claude Skills externalize both agent behavior definitions (SKILL.md) and inter-agent data transfer into structured files, naturally achieving all four principles above.
🎛️ solve_coding_test (Supervisor) 🧩 get_coding_problem 📝 plan_coding_problem 💻 write_coding_problem 🛡️ validate_coding_problem 🔍 reflect_coding_problem

Skills Breakdown

What is a Skill? — A structured markdown file that defines what each agent should do (SKILL.md). When an agent is invoked, the Skill's content is injected into that agent's context to guide its behavior. Unlike prompt instructions, Skills are externalized as files — keeping the main conversation context clean is the key.
🎛️
solve_coding_test Supervisor
The top-level Skill that orchestrates the entire solving loop. problem.md exists, then manages the Plan → Write → Validate → Wait for Judge → Reflect loop from trial 1 up to 10 attempts. Key rule: Every sub-Skill invocation must be dispatched via fresh subagent (Task tool) — never reusing previous sub-agents, creating a new agent for each trial even for the same Skill, keeping the main context small. Judging is human-in-the-loop: request the user to submit, then wait for the verdict (Pass/WA/TLE/MLE/RE).
🧩
get_coding_problem Problem Fetch
Uses Playwright MCP to navigate the SWEA website and explore problem pages. Extracts the problem description, I/O format, constraints, and sample test cases, saving them as problem.md. The next agent only needs to read this file.
📝
plan_coding_problem Plan
problem.md and designs the algorithm. Never writes code — only organizes the thought process: problem analysis → key observations → algorithm selection with justification → pseudocode-level steps → edge cases → complexity analysis, saved as plan_{trial}.md. Also verifies feasibility against PyPy 3.7 time/memory constraints.
💻
write_coding_problem Write
Reads problem.md and plan_{trial}.md, then implements the Python 3.7 (PyPy) code. Faithfully follows the plan's algorithm while conforming to the SWEA I/O pattern (T = int(input()) loop). Saves the completed code as solution_{trial}.py, then runs sample tests and iterates fixes until they pass.
🛡️
validate_coding_problem Validate
Statically validates code from the Write agent before submission. Checks banned imports(sys, io, os), output format, Python 3.7 syntax compatibility, etc. to prevent RE/WA before submission. An intermediate step added in the 3rd architecture.
🔍
reflect_coding_problem Reflect
Triggered on judge failure. problem.md and the full history of all previous plans and solutions to avoid repeating mistakes. Adjusts diagnostic focus by verdict type (WA/TLE/MLE/RE), analyzing what went wrong → why → what the previous plan missed, then generates a revised plan_{next_trial}.md. Does not write code.
1st → 2nd Lesson
"Separate"
After single agent failure, separated into sub-agents. Defined agents with prompt instructions and transferred info via summaries, but fine-grained algorithmic details were lost.
2nd → 3rd Lesson
"Switch to Skills"
Replaced prompt instructions with Claude Skills (SKILL.md). Switched both agent behavior definitions and inter-agent info transfer to file-based, simultaneously solving context rot and information loss — the essence of context engineering.

If prompt engineering is the question of "what to tell the model,"
then context engineering is the question of "how to structure the information provided to the model."

Defining agents via prompt instructions pollutes the context with the instructions themselves. Claude Skills externalize agent behavior definitions into structured files, fundamentally solving this problem.

Even with the same model, same problems, and same Reasoning Effort, changing only the context structure fundamentally changes the results. Passing all 10/10 problems including D7(VH) difficulty is empirical proof of this principle.

Problem Details

Detailed Problem Solutions

#23009 / VH2505 · D7(VH) · Acc. 6.67%
Count of Consecutive Overlapping Intervals
Substitution Decomposition Tree + O(N log N)

Solution Approach

  • Consecutive intervals of permutations structured as a substitution decomposition tree
  • Crossing pairs occur only at linear nodes (ascending/descending)
  • O(k log k) computation via prefix sums of child sizes + binary search at each linear node
Trial 1 — MLE (10/12 TC)
Cause: During linear node flattening in the node pool architecture, old child lists remained unreleased in the pool. Worst case: O(N²) references accumulated in pool_children → exceeded 256MB.
Trial 2 — Passed ✓
Fix: Switched to inline processing. When absorbing linear nodes into parent, immediately compute contribution and discard children_sizes (finalize-on-absorption). Maintains O(N) total memory.
#24424 / VH2521 · D6(VH) · Acc. 23.53%
Correct Path
CFL-Reachability Dijkstra

Solution Approach

  • Context-free language constrained shortest path on a CNF grammar and directed graph
  • State: (nonterminal X, source u, dest v) — max 26³ = 17,576 states
  • Terminal rules seed the heap with distance-1 states; binary rules join on state confirmation
Trial 1 — Passed ✓
State space of 17,576 is small enough for PyPy. Passed on first attempt.
#24529 / VH2525 · D6(VH) · Acc. 7.69%
Snail on a Grid
Greedy + Bigint Bitset Knapsack DP

Solution Approach

  • Place the two smallest values at mandatory waypoints (1,1) and (2,N)
  • Split remaining 2(N-1) values into two groups F (top row), L (bottom row) → minimize sum difference
  • F ascending, L descending → mountain arrangement guarantees window sums
  • Balanced partition via Python bigint bitset knapsack DP
Trial 1 — WA (18/47 TC)
Cause: L group sorted ascending → smallest values from both sides concentrated at transition windows.
Trial 2 — Passed ✓
Fix: L.sort(reverse=True) — one-line fix.
#24702 / VH2530 · D7(VH) · Acc. 30.00%
Powers of 2
p-adic Lifting Discrete Log

Solution Approach

  • Construct target value T via CRT to embed X in the last K digits of f(N) = 2^N mod 10^K
  • 2 is a primitive root mod 5^K → discrete log always exists
  • Hensel-style p-adic lifting extracts the log in O(K) steps
Trial 1 — RE
Cause: import io banned in SWEA environment.
Trial 2 — WA (4,400/12,190 TC)
Cause: With K=71, N values grew to ~10^49, likely exceeding the judge's 64-bit integer range.
Trial 3 — Passed ✓
Fix: Reduced K from 71 to 20. Max N ~1.53×10^14, fitting within 64-bit range.
#25333 / VH2533 · D7(VH) · Acc. 33.33%
Synergy Effect
Meet-in-the-middle + Matrix-Tree

Solution Approach

  • Compute synergy based on "isolation" of meaningful nodes in N-node labeled trees
  • g(t): Closed-form via equi-partition Laplacian
  • Inclusion-exclusion to convert "exactly isolated" → "at least isolated"
  • Meet-in-the-middle for weighted subset-sum count (m ≤ 40)
Trial 1 — TLE
Cause: Iterating all B-subsets for each (r, rA, rB) combination → ~10^8-10^9 iterations.
Trial 2 — Passed ✓
Fix: Unified g(t)×H(t)×N_r into a single weighted subset-sum query. Suffix weighted-sum array reduces to one binary search per B-subset.
#25478 / VH2537 · D7(VH) · Acc. 7.41%
Receiver Assignment
Proper Interval Graph + Interval DP

Solution Approach

  • Place N receivers in X slots; communicate if distance ≤ D — count valid placements
  • Proper interval graph structure + LBFS+ canonical ordering extraction
  • Interval DP with O(2^D) states to count placements per component
  • Twin class symmetry handling, then cross-component convolution for the final answer
Trial 1 — MLE
Cause: 2^N bitmask DP with N=50 → 2^50 states → memory exceeded.
Trial 2 — WA (93/140 TC)
Cause: Simple LBFS failed to guarantee a valid proper interval ordering.
Trial 3 — WA (139/140 TC)
Cause: LBFS+ tiebreaking failed for N≥20. Ordering verification passed but interval DP constraints unsatisfied.
Trial 4 — Passed ✓
Fix: Implemented priority-based LBFS+ with refined partition refinement. Passed 4,211 exhaustive stress tests.
#25660 / VH2540 · D7(VH) · Acc. 4.49%
Santa's Gifts
Popular Matching + Hopcroft-Karp

Solution Approach

  • Assign N gifts to N children — find a "popular matching" where no majority prefers another assignment
  • Two-phase matching: first-choice gift group (A) and non-S1 top gift group (B)
  • Deterministic cascade adds conflicting gifts to S1 simultaneously, preventing exponential branching
  • Hopcroft-Karp bipartite matching to verify S1 coverage
Trial 1 — WA (0/77 TC)
Cause: Exponential backtracking explosion. When k children share the same first choice, 2^k branches occur.
Trial 2 — WA (0/77 TC · output format error)
Cause: Incorrect #T prefix included.
Trial 2 (fix) — Passed ✓
Fix: Deterministic cascade + bipartite matching + output format fix → 77/77 passed.
#25699 / VH2543 · D7(VH) · Acc. 16.67%
Tiling
Lattice Periodicity + Hermite Normal Form

Solution Approach

  • Given H×W observation window from translational tiling of a single polyomino → find minimum tile area
  • Tiling corresponds to a 2D integer lattice; tile area = lattice determinant
  • HNF basis (a,0), (c,d) with O(H²W²) precomputed valid translation table
  • Sequential enumeration from area 1, return first valid HNF
Trial 1 — Passed ✓
Solved this 16.67% acceptance rate problem on the first attempt.
#25841 / VH2601 · D7(VH) · Acc. 25.53%
Sister City Pairing
Kruskal Reconstruction Tree + LCA

Solution Approach

  • Two vehicles moving simultaneously in opposite directions on a weighted graph — analyze swap conditions
  • "Parking space" for yielding exists when the connected component is not a simple path
  • Kruskal reconstruction tree tracks non-path events, subtree_min_np propagated bottom-up
  • Query (X,Y) finds LCA and returns ans_weight(LCA)
Trial 1 — RE
Cause: import io, import os banned in SWEA environment.
Trial 2 — Passed ✓
Fix: Replaced with input() / print(). No algorithm changes.
#26013 / VH2602 · D7(VH) · Acc. 25.00%
Bishop
Ferrers Board Factorization + NTT

Solution Approach

  • Count non-attacking bishop placements on N×N board for k=1..2N-1 (mod 998244353)
  • Diagonal parity decomposition into two independent groups → each becomes a Ferrers board rook placement problem
  • Ferrers board factorization theorem decomposes the rook polynomial into a product of linear factors
  • Finite differences + NTT convolution for O(N log N) computation
Trial 1 — TLE
Cause: O(N²) DP. ~10^10 operations at N=100,000, exceeding the 30-second limit.
Trial 2 — Passed ✓
Fix: Ferrers board factorization + NTT for O(N log N) optimization.
